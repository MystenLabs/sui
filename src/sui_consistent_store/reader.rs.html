4//! The [`Reader`] trait abstracting the read context a
5//! [`DbMap`](crate::DbMap) is bound to.
6//!
7//! The crate ships exactly two reader types: [`Db`] (live-tip) and
8//! [`Snapshot`](crate::Snapshot) (snapshot-bound), each with a
9//! blanket impl on its `&` form for the borrowed variant. There is
10//! no separate `Live`/`LiveRef` wrapper — a [`Db`] handle *is* the
11//! live-tip read context, and a borrow of one *is* the borrowed
12//! live-tip context.
13//!
14//! Every [`DbMap<K, V, R>`](crate::DbMap) is parameterized by a
15//! [`Reader`]. The default is [`Db`], so today's call sites
16//! (`DbMap::new(db, "items")`, `Db::open::<MySchema>(...)`) keep
17//! working unchanged. Re-projecting a schema or a single map at a
18//! captured snapshot — via
19//! [`SchemaAtSnapshot::at`](crate::SchemaAtSnapshot::at) or
20//! [`DbMap::at`](crate::DbMap::at) — produces a parallel handle whose
21//! reader is [`Snapshot`](crate::Snapshot).
22//!
23//! # Why this exists
24//!
25//! The crate previously routed snapshot reads through an entirely
26//! separate type (the snapshot handle's read methods, plus a
27//! borrowed `SnapshotView`). That kept the read API duplicated and
28//! forced call sites to choose between `map.get(&k)?` (live) and
29//! `snap.get(&map, &k)?` (snapshot). With the reader generic, both
30//! call sites read identically: the choice of consistency context is
31//! made once, at the point the schema or map is bound, and from
32//! there every method call is `map.get(&k)?` against whatever reader
33//! is in scope.
34//!
35//! # Cost model
36//!
37//! Constructing a [`DbMap`](crate::DbMap) bound to a [`Db`] costs
38//! one `Arc` bump (the [`Db`] clone). Constructing one bound to
39//! [`Snapshot`](crate::Snapshot) costs two (the snapshot's own
40//! [`Db`] and `Arc<SnapshotEntry>`). Each call to
41//! [`DbMap::at`](crate::DbMap::at) clones the snapshot (two `Arc`
42//! bumps); the column-family name is a [`&'static str`](prim@str),
43//! so it copies without allocating. Re-projecting an N-CF schema
44//! costs 2N `Arc` bumps and N struct constructions per call to
45//! [`SchemaAtSnapshot::at`](crate::SchemaAtSnapshot::at). For a
46//! per-request handler that projects once and reads many times,
47//! this is amortized; for a hot path that projects on every read,
48//! project once outside the loop.
49//!
50//! `&Db` and `&Snapshot` are the zero-`Arc`-bump variants. Use them
51//! via `DbMap::new(&db, "cf")` or
52//! [`DbMap::at_ref`](crate::DbMap::at_ref) when the returned
53//! [`DbMap`](crate::DbMap) is scoped to a single function body and
54//! can be tied to a [`Db`] or [`Snapshot`](crate::Snapshot) the
55//! caller already holds.

61/// Abstracts the read context a [`DbMap`](crate::DbMap) is bound to.
62///
63/// Every implementation supplies (1) the [`Db`] handle needed to
64/// look up the column-family handle and (2) a fresh [`ReadOptions`]
65/// tuned for the reader's consistency context. [`Db`] (and its
66/// `&Db` blanket impl) returns [`ReadOptions::default()`];
67/// [`Snapshot`](crate::Snapshot) (and its `&Snapshot` blanket impl)
68/// returns one with [`set_snapshot`](ReadOptions::set_snapshot)
69/// pointed at the captured snapshot.
70///
71/// # Sealed
72///
73/// The crate ships two reader types — [`Db`] and
74/// [`Snapshot`](crate::Snapshot) — each with a blanket impl on its
75/// `&` form. The trait is sealed via a pub(crate) supertrait so
76/// downstream code cannot add another — a custom reader could
77/// return [`ReadOptions`] referencing a snapshot pointer not
78/// co-owned through the [`Db`] handle story, leading to UB inside
79/// RocksDB.
80pub trait Reader: sealed::Sealed {
81    /// The shared database handle the column family lives on.
82    fn db(&self) -> &Db;
83
84    /// Construct a fresh [`ReadOptions`] configured for this reader.
85    ///
86    /// Implementations are expected to be cheap; the returned options
87    /// are consumed by a single read and dropped. Callers that issue
88    /// many reads in a tight loop pay one fresh allocation per call,
89    /// which matches RocksDB's expected pattern.
90    fn read_options(&self) -> ReadOptions;
91}
92
93pub(crate) mod sealed {
94    pub trait Sealed {}
95}
96
97impl sealed::Sealed for Db {}
98
99impl Reader for Db {
100    fn db(&self) -> &Db {
101        self
102    }
103
104    fn read_options(&self) -> ReadOptions {
105        ReadOptions::default()
106    }
107}
108
109impl sealed::Sealed for &Db {}
110
111impl Reader for &Db {
112    fn db(&self) -> &Db {
113        self
114    }
115
116    fn read_options(&self) -> ReadOptions {
117        ReadOptions::default()
118    }
